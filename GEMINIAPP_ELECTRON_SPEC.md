# GeminiApp for Windows on Electron

## Цель приложения

`GeminiApp` — это настольное Windows-приложение на `Electron`, которое визуально и по UX напоминает приложение `Codex` от OpenAI, но работает поверх `Google Gemini CLI`. Приложение должно выглядеть как полноценный desktop-клиент: с левым сайдбаром, центральной рабочей областью, историей чатов, проектным контекстом, авторизацией через Google-аккаунт, статусами CLI-сессии и стабильным потоком сообщений между пользователем и Gemini.

Основная идея: взять сильные стороны интерфейса Codex-приложения и адаптировать их под сценарий работы с `Gemini CLI`, сохранив ощущение профессионального инструмента для разработки, работы с файлами, проектами и длинными диалогами.

## Общий визуальный стиль

### Базовое впечатление

Интерфейс должен выглядеть:

- строго;
- темно;
- минималистично;
- профессионально;
- аккуратно выровненно;
- без ярких декоративных перегрузок;
- как desktop-инструмент для инженеров, а не как обычный мессенджер.

### Визуальные принципы

- Основная тема: глубокий графитово-черный интерфейс.
- Контейнеры слегка отличаются по тону, без резких контрастов.
- Скругления умеренные: `10px`, `12px`, `14px`, `16px`.
- Границы почти незаметные, чаще через полупрозрачные линии.
- Акценты использовать очень дозированно.
- Важные действия подсвечивать светлым текстом, а не кислотными цветами.
- Интерфейс должен ощущаться “дорогим” и спокойным.

### Типографика

Для Windows лучше использовать одну из связок:

- `Segoe UI Variable` как нативный системный вариант;
- `Inter` для контента и интерфейсных блоков;
- `JetBrains Mono` для терминала, логов, путей и CLI-вывода.

Рекомендуемое разделение:

- главное UI: `Inter` или `Segoe UI Variable`;
- терминальные блоки и команды: `JetBrains Mono`.

### Плотность интерфейса

Интерфейс должен быть визуально плотным, но не тесным:

- высота строки в списках: `36px`–`44px`;
- внутренние отступы карточек: `12px`–`16px`;
- ширина левого сайдбара: `320px`–`380px`;
- максимальная ширина центрального диалогового контейнера: `980px`–`1180px`.

## Архитектура окна приложения

### Окно Windows

Приложение должно быть именно desktop-first:

- одно главное окно;
- кастомная верхняя панель;
- поддержка стандартных кнопок Windows: свернуть, развернуть, закрыть;
- тёмная рамка без яркого системного фона;
- при желании: `titleBarStyle: hidden` + собственный draggable header.

### Основные зоны интерфейса

Окно делится на 3 большие области:

1. Верхняя панель приложения.
2. Левый сайдбар с навигацией и историей.
3. Центральная рабочая область чата.

Опционально справа может появляться:

- панель контекста проекта;
- панель сведений о сессии;
- панель предпросмотра файлов;
- панель логов CLI.

## Верхняя панель

### Назначение

Верхняя панель должна напоминать Codex по ощущению: тонкая, спокойная, почти незаметная, но с важными действиями под рукой.

### Содержимое слева

- кнопка collapse/expand сайдбара;
- назад;
- вперед;
- меню приложения: `Файл`, `Правка`, `Вид`, `Окно`, `Справка`.

### Содержимое по центру

- активный workspace/project path;
- имя текущего чата;
- индикатор текущего режима Gemini;
- опционально строка поиска.

### Содержимое справа

- индикатор авторизации Google;
- аватар пользователя;
- статус подключения к Gemini CLI;
- селектор модели;
- кнопка быстрых настроек;
- кнопка открытия профиля/аккаунта.

### Поведение

- хедер не должен визуально спорить с чатом;
- hover-состояния очень мягкие;
- активные элементы подсвечиваются лишь немного;
- текущий аккаунт и модель должны читаться с первого взгляда.

## Левый сайдбар

### Назначение

Сайдбар — это основная навигация, история чатов и проекты, как в Codex.

### Верхний блок навигации

Содержимое:

- `Новый чат`
- `Поиск`
- `Проекты`
- `Расширения` или `Инструменты`
- `Автоматизации` или `Задачи`

Каждый элемент:

- иконка слева;
- текст справа;
- крупная кликабельная зона;
- hover с легким осветлением фона.

### Блок проектов

Ниже навигации идет список проектов/workspaces:

- имя проекта;
- иконка папки;
- признак активного проекта;
- количество недавних чатов;
- при необходимости badge статуса.

Примеры:

- `Lab`
- `SWebAPI`
- `CurseProj3`

### Вложенный список чатов

Под каждым проектом:

- список последних чатов;
- заголовок чата обрезается многоточием;
- справа время последнего обновления;
- текущий чат выделяется фоном;
- hover показывает меню из трех точек.

### Нижняя часть сайдбара

Внизу фиксированно:

- `Настройки`
- `Аккаунт`
- `Помощь`
- индикатор версии приложения

### Состояния

Сайдбар должен поддерживать:

- обычный режим;
- узкий режим только с иконками;
- resize пользователем;
- скрытие на малой ширине окна.

## Центральная область чата

### Общая композиция

Центральная область состоит из:

1. Пустого стартового состояния.
2. Ленты сообщений.
3. Композитора ввода.
4. Нижней строки контекста проекта/режима.

### Пустое состояние

До первого сообщения экран должен быть очень похож по композиции на Codex:

- много свободного пространства;
- в центре большой вопрос-заголовок;
- ниже крупный input-контейнер;
- минимум визуального шума.

Пример текста:

`Над чем нужно поработать в GeminiApp?`

Дополнительно можно показать:

- последние действия;
- шаблоны промптов;
- быстрые команды;
- список недавно открытых проектов.

### Лента сообщений

После начала чата в центре появляется прокручиваемая область сообщений.

Сообщения пользователя:

- выравнивание по правому или центральному контейнеру;
- тёмно-серый bubble;
- белый текст;
- минимум декоративности.

Сообщения Gemini:

- ближе к левому краю контентной колонки;
- более “плоская” подача, не как мессенджер;
- поддержка Markdown;
- кодовые блоки;
- списки;
- таблицы;
- inline code;
- статус потока генерации.

### Визуальное поведение ответов

Ответ Gemini должен поддерживать:

- streaming текста по мере генерации;
- блок “думает/выполняет”;
- лог этапов;
- вставки терминального вывода;
- разворачиваемые tool-calls;
- кнопки `Повторить`, `Скопировать`, `Остановить`, `Продолжить`.

### Сообщения инструментов

Если Gemini CLI работает с командами, файлами и действиями, их нужно показывать отдельными блоками:

- `Выполняется команда`
- `Чтение файла`
- `Изменение файла`
- `Ошибка`
- `Успешно`

Эти блоки должны напоминать activity stream Codex:

- компактные;
- моноширинные;
- с иконками состояния;
- по клику раскрывается полный лог.

## Композитор ввода

### Основной контейнер

Поле ввода внизу центральной области — один из ключевых визуальных элементов.

Оно должно быть:

- широким;
- округлым;
- слегка приподнятым на фоне;
- визуально тяжелее остальных контролов;
- как главная точка взаимодействия.

### Состав input-блока

Слева:

- кнопка добавления файла;
- кнопка добавления изображения;
- кнопка выбора контекста проекта.

В центре:

- многострочное поле ввода;
- placeholder в стиле Codex;
- поддержка вставки больших промптов;
- поддержка drag-and-drop файлов.

Справа:

- выбор режима Gemini;
- выбор модели;
- кнопка голосового ввода, если нужна;
- большая кнопка отправки.

### Нижняя техническая строка

Под главным input-контейнером:

- активный проект;
- рабочая папка;
- sandbox mode;
- доступ к инструментам;
- текущий профиль Gemini CLI.

Пример:

`geminiapp · D:\prog\geminiapp · Gemini CLI connected`

### Поведение

- Enter отправляет;
- Shift+Enter переносит строку;
- при генерации кнопка отправки превращается в stop;
- input растет по высоте до разумного лимита;
- при ошибке CLI под input показывается компактный alert.

## Экран авторизации

### Главная задача

Так как приложение должно использовать аккаунт Google, первый вход должен быть понятным и нативным.

### Отдельный onboarding-экран

При первом запуске вместо чата показывается onboarding:

- логотип GeminiApp;
- краткое описание;
- кнопка `Войти через Google`;
- подпись про безопасную авторизацию;
- статус готовности Gemini CLI.

### Визуальный стиль

Экран входа не должен быть ярким. Он должен сохранять общий темный стиль:

- тёмный фон;
- крупный заголовок;
- светлая кнопка входа;
- аккуратная карточка по центру;
- тонкие пояснения ниже.

### Поток Google OAuth

Рекомендуемый поток:

1. Пользователь нажимает `Войти через Google`.
2. Открывается системное OAuth-окно или встроенный защищенный браузер.
3. Пользователь выбирает Google-аккаунт.
4. Приложение получает `authorization code`.
5. Бэкенд приложения меняет его на `access token` и `refresh token`.
6. Токены сохраняются в защищенное хранилище Windows.
7. После успеха открывается основной интерфейс.

### Хранение токенов

Использовать:

- `keytar` для secure storage;
- либо `safeStorage` + шифрование и хранение в профиле пользователя;
- refresh token не хранить в открытом виде в json без шифрования.

### Что показывать после входа

После успешной авторизации в правом верхнем углу:

- аватар Google;
- имя пользователя;
- email;
- меню `Выйти`, `Сменить аккаунт`, `Управление доступом`.

## Сценарий работы с Gemini CLI

### Общая модель взаимодействия

Приложение не должно пытаться выглядеть как чисто web-чат. Оно должно явно показывать, что под капотом работает `Gemini CLI`.

Архитектурная идея:

- UI на `Electron + React`;
- main process управляет жизненным циклом окна и безопасностью;
- preload слой открывает контролируемый IPC API;
- backend bridge запускает `Gemini CLI` как дочерний процесс;
- renderer получает поток stdout/stderr и отображает его как шаги ответа.

### Что должен уметь bridge-слой

- запуск новой CLI-сессии;
- отправка промпта в активную сессию;
- отмена генерации;
- чтение stdout;
- чтение stderr;
- обработка exit code;
- переключение workspace;
- передача переменных окружения;
- ограничение команд и sandbox-режимов.

### Вариант визуализации CLI

Пользователь не должен видеть “сырой терминал” всегда. Лучше смешанный режим:

- основной ответ показывается как chat message;
- промежуточные шаги отображаются как tool/activity blocks;
- полный raw log открывается в collapsible panel;
- при желании можно переключиться в `Terminal View`.

### Статусы Gemini CLI

В приложении нужен заметный, но аккуратный статус:

- `Connected`
- `Starting`
- `Authorizing`
- `Busy`
- `Streaming`
- `Stopped`
- `Error`

Индикация:

- маленький цветной индикатор;
- текст рядом;
- hover показывает подробности.

## Поведение чата

### Создание нового чата

При нажатии `Новый чат`:

- очищается текущая лента;
- создается новая session entity;
- сбрасывается контекст ответа;
- сохраняется текущий workspace;
- фокус ставится в input.

### Отправка сообщения

1. Пользователь вводит промпт.
2. Сообщение сразу появляется в ленте.
3. UI показывает состояние `Gemini thinking`.
4. Запускается вызов в bridge к CLI.
5. Поток ответа постепенно рисуется в сообщении ассистента.
6. Отдельно идут tool/activity записи.
7. После завершения фиксируется итоговый ответ и служебные метаданные.

### Повторная генерация

Нужна кнопка:

- `Regenerate`

Она:

- повторно использует предыдущий prompt;
- создает новый assistant response;
- сохраняет историю попыток.

### Ошибки

Если CLI не ответил или завершился с ошибкой:

- показать alert-плашку внутри чата;
- дать кнопку `Повторить`;
- дать кнопку `Открыть лог`;
- не ломать весь интерфейс.

## Проекты и рабочие пространства

### Идея

Как и в Codex, приложение должно быть ориентировано на работу с конкретными папками/репозиториями.

### Для каждого workspace хранить

- уникальный id;
- имя;
- абсолютный путь;
- дату последнего открытия;
- связанные чаты;
- предпочитаемую модель;
- пользовательские настройки запуска CLI.

### Поведение

- переключение проекта меняет текущий контекст;
- новый чат можно создавать в рамках конкретного проекта;
- путь проекта всегда виден в UI;
- при недоступном пути показывается предупреждение.

- `Dark Contrast`
- `System`

Базовая тема по умолчанию должна быть именно в стиле скриншота: тёмная, спокойная, низкоконтрастная, с лёгкими серыми переходами.

---

## Анимации и микровзаимодействия

Анимации должны быть аккуратными:

- плавное появление сообщений;
- fade-in для стартового экрана;
- мягкий hover у списка чатов;
- плавное изменение высоты input;
- деликатный shimmer или cursor-blink при streaming.

Нельзя делать:

- резкие spring-анимации;
- яркие вспышки;
- слишком “игровой” интерфейс.

---

## UX-детали, которые дадут ощущение Codex-приложения

### Важные акценты

- много воздуха в центре;
- темный, почти бесшовный фон;
- крупная строка ввода как ядро экрана;
- история чатов в левом столбце;
- проектно-ориентированная работа;
- ощущение, что под капотом агент/CLI, а не просто чат-бот.

### Что особенно важно реализовать визуально

- широкий центральный холст;
- приглушенная палитра;
- аккуратные скругления;
- легкие границы;
- компактный верхний бар;
- спокойные иконки `lucide-react` или похожего набора;
- моноширинные вставки для путей, логов и команд.

---

## Безопасность и технические ограничения

### Для Electron обязательно

- `contextIsolation: true`
- `nodeIntegration: false`
- все системные операции только через preload/API
- whitelisted IPC channels
- строгая валидация payload

### Для OAuth

- использовать PKCE;
- не хранить чувствительные данные в renderer;
- токены обновлять в main process;
- по возможности использовать системный браузер для логина.

### Для CLI

- ограничить исполняемые команды;
- логировать ошибки отдельно;
- отделять stdout от stderr;
- корректно завершать дочерние процессы при закрытии окна.

---

## Рекомендации по Windows-нативности

Чтобы приложение ощущалось нативным под Windows:

- использовать системные клавиатурные сочетания;
- поддержать `Acrylic/Mica` только если это не ломает читаемость;
- правильно обработать масштабирование `125%`, `150%`;
- сделать качественный tray icon опционально;
- поддержать уведомления Windows;
- использовать привычные системные диалоги выбора папки.

## Как должен выглядеть “успешный чат”

Успешный сценарий использования должен выглядеть так:

1. Пользователь открывает приложение.
2. При необходимости логинится через Google.
3. Выбирает проект или рабочую папку.
4. Видит пустой холст с крупным input.
5. Пишет задачу для Gemini.
6. Ответ начинает стримиться как в умном IDE-ассистенте.
7. Ниже или внутри ответа отображаются шаги CLI.
8. Пользователь видит, что запрос выполнен, файлы прочитаны, команды отработали.
9. История автоматически сохраняется в левом сайдбаре.
10. При повторном открытии приложения чат и проектный контекст восстанавливаются.

Если этот сценарий реализован плавно, визуально чисто и стабильно, приложение действительно будет ощущаться как “Codex для Gemini CLI”.

---

## Итоговая концепция

`GeminiApp` должен быть не просто оболочкой над CLI, а цельным desktop-продуктом:

- с интерфейсом уровня современного инженерного инструмента;
- с сильной проектной навигацией;
- с авторизацией через Google;
- с чатом, который поддерживает потоковые ответы и шаги агента;
- с дизайном, близким к Codex по настроению, плотности и композиции;
- с явной интеграцией `Gemini CLI`, но без ощущения “голого терминала”.

Главная цель внешнего вида: пользователь должен открыть приложение и сразу понять, что это серьезный рабочий AI-клиент для Gemini, выполненный в стиле настольного Codex-интерфейса.
